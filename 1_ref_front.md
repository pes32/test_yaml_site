# План: полный переход Vue UI на `.vue` SFC и упорядочивание загрузки виджетов

## 1. Цель

Перевести текущий Vue UI на `.vue` Single File Components без изменения публичного серверного и виджетного контракта, убрать зависимость от inline `template:` как основной модели, привести входные точки `page` и `debug` к нормальной SFC-схеме и упорядочить загрузку тяжёлых виджетов через асинхронный реестр и разбиение на чанки.

Дополнительная цель — уменьшить initial bundle страницы и исключить попадание тяжёлых семейств виджетов в стартовую загрузку без необходимости.

Критерием успеха является не только уменьшение размера сборки, но и отсутствие регресса по времени первой интерактивности, по сетевому пути первого экрана и по базовым пользовательским сценариям после миграции.

## 2. Что входит в задачу

### 2.1. Подготовительный аудит: зафиксированное текущее состояние

#### 2.1.1. Точки входа и build-контур
- `tooling/vite/src/entry/page.ts` и `tooling/vite/src/entry/debug.ts` сейчас являются тонкими entrypoints и только импортируют legacy-модули `frontend/js/page.js` и `frontend/js/debug.js`.
- Корневая Vue-разметка пока не вынесена в SFC и живёт в серверных шаблонах `templates/page.html` и `templates/debug.html`.
- `tooling/vite/vite.config.ts` содержит только два entrypoint (`page`, `debug`), но не содержит `@vitejs/plugin-vue` и не содержит `manualChunks`.
- Текущий `typecheck` в `tooling/vite/package.json` — это `tsc --noEmit`.
- Текущий `tooling/vite/tsconfig.json` проверяет только `src/entry/*` и `src/contracts/*`; widget-layer в `frontend/js/**` сейчас не покрыт типизацией как migration target.

#### 2.1.2. Инвентаризация логических Vue-компонентов
- Root apps: `page` в `frontend/js/page.js` + `templates/page.html`, `debug` в `frontend/js/debug.js` + `templates/debug.html`.
- Root/common UI-компоненты: `WidgetRenderer`, `ItemIcon`, `ContentRows`, `SectionCard`, `ModalButtons` из `frontend/js/widgets.js`; `ModalManager`; `ConfirmModal`; `DiagnosticsPanel`; `ErrorPanel`; `Md3Field`.
- Production widget-компоненты из реестра: `str`, `int`, `float`, `text`, `img`, `button`, `ip`, `ip_mask`, `list`, `voc`, `split_button`, `date`, `time`, `datetime`, `table`.
- Тонкие re-export модули `frontend/js/widgets/voc.js`, `frontend/js/widgets/datetime_widgets.js`, `frontend/js/widgets/table/index.js` не являются самостоятельными migration targets и должны либо исчезнуть, либо остаться только как совместимые import-адаптеры.
- Все 15 типов из текущего `DEFAULT_WIDGET_REGISTRY` реально используются в `pages/**/*.yaml`; мёртвых widget types в реестре сейчас нет.

#### 2.1.3. Фактическое использование виджетов в YAML
- По репозиторию `pages/**/*.yaml` сейчас зафиксированы следующие частоты использования: `button` — 41 вхождение в 7 файлах, `table` — 34 вхождения в 14 файлах, `text` — 16 вхождений в 5 файлах, `date` — 14 вхождений в 8 файлах, `list` — 14 вхождений в 4 файлах.
- Остальные типы используются так: `int` — 7/4, `float` — 5/2, `ip_mask` — 5/4, `img` — 4/2, `split_button` — 4/2, `voc` — 4/4, `time` — 4/4, `str` — 3/3, `ip` — 3/2, `datetime` — 2/2.
- Практический вывод: `table` тяжёлый, но уже широко размазан по страницам; `date` тоже частый; `voc` и `split_button` встречаются заметно локальнее. Это важно для порядка миграции и для async-chunk policy.

#### 2.1.4. Текущий widget registry и глобальная регистрация
- Центральный widget registry живёт в `frontend/js/widgets/factory.js`, eagerly импортирует все 15 widget-компонентов и регистрирует их в `Map` при загрузке модуля.
- `WidgetRenderer` в `frontend/js/widgets.js` разрешает компонент через `widgetFactory.getWidgetComponent(widgetType)` и fallback'ится в `str`, если тип неизвестен.
- `TableWidget` уже сейчас является скрытым обходом центрального registry: он напрямую импортирует `DateTimeWidget`, `DateWidget`, `TimeWidget`, `FloatWidget`, `IntWidget`, `IpWidget`, `IpMaskWidget`, `ListWidget`, `StringWidget`, `VocWidget`.
- Следствие: текущая архитектура имеет не один, а полтора реестра. Формально source of truth — `widgetFactory`, но `table` держит собственный встроенный набор cell editors и тем самым создаёт скрытый coupling для chunking.
- В `frontend/js/page.js` есть глобальная регистрация shared/root-компонентов через `app.component(...)`: `widget-renderer`, `item-icon`, `section-card`, `content-rows`, `modal-manager`, `modal-buttons`, `confirm-modal`, `error-panel`.
- В `frontend/js/debug.js` есть своя глобальная регистрация: `diagnostics-panel`, `error-panel`.
- Глобальной регистрации field widgets через `app.component(...)` сейчас нет; они попадают в UI только через `WidgetRenderer + widgetFactory`.
- Глобальных `app.mixin(...)` сейчас нет; mixin-слой существует только на уровне компонентов.

#### 2.1.5. Mixins и текущая модель state ownership
- Базовый `widgetMixin` используется в `str`, `int`, `float`, `text`, `list`, `ip`, `ip_mask`, `date`, `time`, `datetime` и косвенно через `vocComponentMixin`.
- `dateTimeMixin` — отдельный слой общей логики для `date` / `time` / `datetime`.
- `vocComponentMixin` уже сейчас является стеком `widgetMixin + vocModalMixin`, то есть `voc` — это не простой компонент, а многослойный mixin runtime.
- `runtime/widget_contract.js` считает stateful-типами только `str`, `text`, `int`, `float`, `date`, `time`, `datetime`, `ip`, `ip_mask`, `list`, `voc`.
- `button`, `split_button`, `img` и `table` не входят в generic stateful-widget path. У `table` отдельное внутреннее состояние через `table_api.createStore`, поэтому перенос state ownership у таблицы нельзя смешивать с общей логикой простых полей.

#### 2.1.6. Текущий `provide/inject`
- `page` root через `provide()` публикует следующие ключи: `getConfirmModal`, `openUiModal`, `closeUiModal`, `getWidgetConfigByName`, `getWidgetAttrsByName`, `getWidgetRuntimeValueByName`, `getModalRuntimeState`, `getModalRuntimeController`, `getCurrentPageNameFromRuntime`, `getAllAttrsMap`, `showAppNotification`, `reportAppError`, `handleRecoverableAppError`, `setActiveDraftWidgetController`, `clearActiveDraftWidgetController`.
- Фактические потребители `inject` сейчас такие: `button` и `split_button` зависят от `getConfirmModal/openUiModal/closeUiModal`; `ModalButtons` и `ModalManager` зависят от `getWidgetAttrsByName/getWidgetRuntimeValueByName`; `ModalManager` зависит от `getModalRuntimeState/getModalRuntimeController`; `widgetMixin` зависит от `showAppNotification/setActiveDraftWidgetController/clearActiveDraftWidgetController`; `TableWidget` зависит от `getAllAttrsMap/handleRecoverableAppError/showAppNotification`.
- В текущем frontend-коде не найдено потребителей для `getWidgetConfigByName`, `getCurrentPageNameFromRuntime` и `reportAppError`; эти ключи существуют в root `provide`, но не образуют фактический runtime-contract между компонентами.
- `debug` root не использует `provide/inject` и живёт как отдельный маленький app.

#### 2.1.7. DOM-чувствительные и lifecycle-чувствительные зоны
- `SectionCard` зависит от измерения высоты через `getBoundingClientRect()`, ручного `transitionend` и конкретной collapse-структуры.
- `ModalManager` зависит от `ref="modalScrollRoot"`, от scroll restore и от структуры секций/табов, приходящей из `modal_runtime_service`.
- `ListWidget` — тяжёлый DOM-чувствительный виджет: `Teleport` в `body`, refs `dropdownRoot/dropdownToggle/dropdownMenu`, click-outside, scroll listeners, focus restore, селекторы `[role="option"]`, `.dropdown-item`, `data-dropdown-open`, `data-table-action-trigger`, `data-table-embedded-widget`.
- `VocWidget` ещё тяжелее: inline dropdown + отдельная modal через `Teleport`, отдельный modal search/sort/active-row state, outside click, scroll listeners, focus restore и свой table-like DOM внутри модалки.
- `SplitButtonWidget` — тяжёлый DOM-sensitive runtime: `Teleport`-menu, вычисление геометрии через `getBoundingClientRect`, глобальные listeners `scroll/resize/click`, traversal по глобальному `FOCUSABLE_SELECTOR`, ручное восстановление фокуса.
- `DateWidget`, `TimeWidget`, `DateTimeWidget` используют `Teleport`, floating-position runtime, outside listeners и ручную синхронизацию focus/picker state.
- `TableWidget` — зона максимального риска: крупный шаблон + 20+ helper/runtime-модулей, sticky-header clone, context menu, selection, keyboard navigation, clipboard, grouping, lazy sentinel, embedded widgets, опора на селекторы `tbody td`, `[data-row]`, `[data-col]`, `[data-table-embedded-widget="true"]`, `[data-table-action-trigger]`, `.widget-table__th-inner`, `.widget-table__cell-value`.
- `IpWidget` / `IpMaskWidget` легче, но тоже не полностью "чистые": есть caret-sensitive ввод и прямой `querySelector('input.widget-ip')`.

#### 2.1.8. Runtime-зависимости, которые уже надо учитывать в плане
- `page` root зависит от `gui_parser`, `api_client`, `attrs_loader`, `bootstrap`, `error_model`, `execute_flow`, `modal_runtime_service`, `page_bootstrap_flow`, `page_selectors`, `page_session_store`, `page_store`.
- `debug` root зависит от `api_client`, `error_model` и `feedback.js`; общий widget runtime он сейчас не использует.
- `button` и `split_button` сидят на `action_runtime` / `action_executor`.
- `list` зависит от `runtime/widget_contract.js`.
- `voc` зависит от `runtime/voc_contract.js` и собственного `voc_*` runtime-слоя.
- `table` зависит от `table_api`, `table_core`, набора `table_*` runtime-модулей и напрямую тянет другие widget-компоненты в cell editor path.

#### 2.1.9. Выводы для порядка миграции
- Первый реальный архитектурный разрез проходит не по виджетам, а по `frontend/js/page.js` / `frontend/js/debug.js` и `templates/page.html` / `templates/debug.html`: пока они не заменены на root SFC, legacy runtime остаётся реальным центром инициализации.
- Первый низкорисковый SFC-wave: `feedback.js`, `confirm_modal.js`, `md3_field.js`, shared-компоненты из `widgets.js`, затем `button`, `img`, `str`, `int`, `float`, `text`, `ip`, `ip_mask`.
- `date` / `time` / `datetime` логично стабилизировать до `table`, потому что текущая таблица напрямую импортирует эти редакторы.
- `voc`, `list` и `split_button` надо считать отдельными DOM-sensitive migration families, а не "обычными полями".
- `table` нельзя считать изолированным family chunk, пока не разорвана его прямая import-зависимость на `datetime/list/voc` и другие cell editors.
- Из-за широкого использования `table` и `date` lazy-loading нельзя проектировать только по размеру файлов; нужно смотреть на реальное присутствие этих виджетов на страницах и на сетевой путь первого экрана.

### 2.2. SFC-миграция
- Подключение полноценной поддержки `.vue` SFC в текущую Vite-схему.
- Перевод root-приложений `page` и `debug` на root SFC.
- Перевод текущих Vue-компонентов на `.vue`.
- Переход на Composition API и `<script setup>` для новых SFC.
- Отказ от inline `template:` как основной модели для production-компонентов.
- Замена mixin-слоя на composables и utility-модули.

### 2.3. Загрузка и бандлинг
- Введение асинхронного реестра виджетов вместо eager widgetFactory.
- Выделение тяжёлых семейств в отдельные async chunks.
- Загрузка тяжёлых виджетов по требованию.
- Ограниченный prefetch только для реально нужных типов виджетов.
- Проверка отсутствия скрытого re-import тяжёлых зависимостей обратно в initial graph.

### 2.4. Проверка результата
- Typecheck и build на новой схеме.
- Smoke-проверка `page` и `debug`.
- Сравнение bundle output до и после рефакторинга.
- Проверка того, что тяжёлые чанки не входят в initial page bundle без необходимости.
- Проверка отсутствия регресса по сетевому пути первого экрана и времени первой интерактивности.

## 3. Что не входит в задачу

- Изменение backend API.
- Изменение серверного bootstrap payload.
- Изменение Jinja header include.
- Изменение публичного YAML-контракта.
- Полный редизайн CSS, layout или theme.
- Переписывание бизнес-логики виджетов вне объёма, необходимого для SFC-миграции и async loading.
- Рефакторинг несвязанных frontend-подсистем.
- Любые изменения Sudoku категорически запрещены. Этот модуль должен остатся отдельным.
- Переименование `props` / `emits` / `inject` / `commands` "для красоты".
- Унификация событийных контрактов между виджетами, если это не требуется непосредственно для миграции.
- Переписывание серверного payload adapter-слоя.

## 4. Жёсткие ограничения

- На каждом этапе проект должен оставаться собираемым и запускаемым.
- Частичная миграция допустима только в том виде, при котором `page` и `debug` продолжают работать.
- Публичный контракт виджетов должен быть сохранён.
- Перфоманс-оптимизация не должна менять пользовательскую механику виджетов.
- Нельзя заменять один неявный runtime-контракт другим неявным контрактом.
- Любая временная совместимость старого и нового слоя должна быть явно локализована, иметь owner и этап удаления.
- Нельзя сохранять legacy frontend runtime как реальный центр инициализации под поверхностью новых SFC.
- Нельзя допускать неявное общее реактивное состояние между разными экземплярами виджетов.
- Нельзя менять каскад CSS и точки подключения CSS как побочный эффект SFC-миграции.
- Миграция в SFC и TypeScript не должна проходить через массовое размазывание `any`.

## 5. Публичный контракт, который должен быть сохранён

### 5.1. На уровне данных и API
Сохраняются без изменения:
- `widgetConfig`
- `widgetName`
- существующие `input` / `execute`
- текущие `provide/inject` ключи
- серверный bootstrap payload
- существующие данные для `page` и `debug`

### 5.2. На уровне поведения
Сохраняются без изменения:
- наблюдаемая runtime-семантика виджетов
- порядок и смысл пользовательских действий
- текущие сценарии открытия/закрытия/редактирования
- focus-flow, keyboard-flow и modal/dropdown lifecycle там, где они уже существуют
- критичный DOM-contract в местах, где на него завязаны CSS, runtime или вспомогательная логика

Публичным контрактом считаются не только `props` / `emits` / `inject`, но и наблюдаемое поведение.

### 5.3. Жёсткий DOM-contract для чувствительных виджетов
Для DOM-чувствительных участков без изменения сохраняются:
- структура ключевых узлов;
- классы и `data-*`-атрибуты, если на них опираются CSS или JS;
- точки фокуса и ожидаемый tab/focus-flow;
- `refs` и селекторы, если на них опирается runtime, измерения или вспомогательная логика;
- порядок вложенности там, где на него завязаны popup/dropdown/table-механики;
- mount/unmount-узлы, если они участвуют в позиционировании, расчётах размеров или управлении overlay.

Формулировка "поведение в целом то же" не считается достаточной, если при этом меняется чувствительный DOM-контракт.

### 5.4. CSS-contract на время миграции
- Существующая глобальная стилизация остаётся основной моделью.
- `scoped styles` не вводятся без отдельного обоснования.
- Существующая глобальная стилизация не переносится стихийно в локальную.
- Каскад, порядок подключения и точки подключения CSS не меняются как часть SFC-миграции.
- Если у компонента появляются локальные стили, они не должны ломать существующий `theme/layout` contract.

### 5.5. Parity между `page` и `debug`
- Shared-модули обязаны работать одинаково в `page` и `debug`.
- `debug` может иметь собственные диагностические зависимости и UI-обвязку, но не отдельную семантику общих виджетов.
- Обратная зависимость `page -> debug` запрещена.
- Запрещена "починка только в debug" или "особая ветка логики только для page" без документированного основания.

## 6. Целевое архитектурное состояние

### 6.1. Root-приложения
- `page` монтирует `PageApp.vue`.
- `debug` монтирует `DebugApp.vue`.

Серверные шаблоны остаются тонкими shell-файлами:
- `head`
- bootstrap JSON
- серверный header include
- контейнер для mount root app

### 6.2. Компоненты
- Production-компоненты Vue лежат в `.vue`.
- Новые компоненты пишутся через Composition API и `<script setup>`.
- Глобальная регистрация компонентов не используется как основная модель.
- Внутри SFC используются явные импорты и локальная композиция.

### 6.3. Владение состоянием и общая логика
- Владельцем widget state по умолчанию является конкретный экземпляр виджета.
- Локальное состояние остаётся внутри компонента, если оно не является частью существующего межкомпонентного контракта.
- Состояние поднимается в `parent/root` только там, где это реально требуется для координации дочерних компонентов в рамках уже существующего поведения.
- Singleton-like модули не должны владеть интерактивным состоянием отдельных виджетов.
- Общее реактивное состояние допустимо только через явную фабрику, composable или `provide/inject` на уровне конкретного root/instance.
- `list`, `voc`, `split_button` и `table` не должны получать неявное общее реактивное состояние между разными экземплярами.
- Mixins заменяются на composables и utility-модули.
- Общие реактивные куски находятся в composables.
- Чистые helper-функции без реактивного состояния находятся в utils.
- Нельзя собирать новый "god-composable", повторяющий проблему mixin-слоя в другой форме.

### 6.4. Реестр виджетов и chunking
- Реестр виджетов становится явным, модульным и типизированным.
- Для тяжёлых семейств используется async loading.
- Для базовых и дешёвых компонентов синхронная загрузка допускается.
- Контракт async registry типизируется как явное соответствие `widget type -> component loader`.
- Chunking делается по границам реального совместного использования, а не "по красоте".
- `list`, `voc`, `split_button` можно держать вместе только если это реально уменьшает initial path и не притягивает лишние зависимости.
- Один тяжёлый helper не должен затягивать весь family chunk обратно в стартовую загрузку.
- `common`, `composables`, `table utils` и прочие shared-модули отдельно проверяются на скрытый re-import тяжёлых зависимостей в initial graph.
- По текущему аудиту `table` является скрытой coupling-point для chunking, потому что напрямую импортирует `datetime/list/voc/ip/int/float/str` как embedded editors; эта связность должна быть разорвана или локализована до финальной оценки async topology.

### 6.5. Legacy bridge и критерии демонтажа legacy runtime
Временным bridge допускаются только локальные совместимые адаптеры следующих типов:
- thin adapter, который временно публикует ещё не перенесённый legacy-виджет через новый реестр;
- локальный wrapper вокруг существующей глобальной регистрации, mixin API или lifecycle-hook, если без него нельзя завершить промежуточный этап;
- совместимый переходник, который прозрачно переводит старый контракт инициализации в явный импортный контракт нового entrypoint.

Граница между допустимым bridge и архитектурным мусором проходит так:
- bridge не владеет глобальной инициализацией, порядком загрузки и source of truth для UI;
- bridge локален, имеет понятную точку входа и понятное удаление;
- bridge не создаёт новое скрытое shared state;
- bridge не используется как обязательный слой для всех новых SFC;
- bridge не подменяет собой новый widget registry, root app или lifecycle runtime.

Архитектурным мусором считается любой слой, который:
- остаётся реальным центром инициализации после появления root SFC;
- управляет последовательностью загрузки через legacy script-order;
- хранит глобальное состояние, от которого фактически зависят новые SFC;
- остаётся источником истины для регистрации виджетов после завершения переноса соответствующего семейства.

Bridge допустим только как переходное решение на этапах 2-5.

К концу этапа 6:
- все временные bridge-модули должны быть либо удалены, либо сведены к нулю в production-path;
- `page` и `debug` должны инициализироваться через новый Vite runtime без sequential legacy-loader;
- запрещена ситуация, в которой SFC находятся сверху, а старый runtime всё ещё остаётся реальным центром инициализации.

### 6.6. TypeScript-contract
- Миграция в SFC не должна превращаться в массовое размазывание `any`.
- Новые composables должны иметь явные типы на публичных входах и выходах.
- Widget registry и async loaders должны иметь типизированный публичный контракт.
- Временные type assertions допустимы только локально, с явным комментарием и только если без них нельзя пройти текущий этап.
- Typecheck должен отражать реальную проверку `*.vue`, а не обходиться через временные исключения или `transpile-only`.

## 7. Переходные правила миграции

### 7.1. Общие правила
- Старые inline-компоненты могут временно сосуществовать с новыми SFC только в переходный период.
- Источник истины для регистрации виджетов на каждом этапе должен быть один.
- Нельзя оставлять дублирующие пути регистрации одного и того же виджета без явного правила приоритета и плана удаления.
- Каждый шаг миграции должен завершаться рабочим состоянием проекта.
- Нельзя совмещать структурный перенос и попутный API cleanup без прямой необходимости для миграции.

### 7.2. Требование к entrypoints
После завершения задачи:
- `page` и `debug` должны быть нормальными Vite entrypoints;
- импортный граф должен определять порядок инициализации;
- ручная script-order магия не должна быть основой frontend runtime;
- любой временный bridge должен подключаться только из явного import graph, а не через отдельный sequential legacy-loader.

### 7.3. Прямые запреты на попутный cleanup
- Нельзя переименовывать `props` / `emits` / `inject` / `commands` "для красоты".
- Нельзя унифицировать событийные контракты разных виджетов, если это не обязательно для миграции.
- Нельзя переписывать серверный payload adapter-слой в рамках этой задачи.

## Этап 6. Замена mixins на composables и удаление legacy bridge
### Цель
Убрать mixin-слой без создания нового неуправляемого слоя общей логики и завершить демонтаж legacy frontend runtime.

### Вынести в composables
- общий widget state только там, где он действительно общий по контракту;
- `datetime`-логику;
- `VOC/modal`-логику;
- общий UI runtime, если он действительно общий и не создаёт скрытый shared state.

### Вынести в utils
- чистые helper-функции;
- преобразования данных;
- функции без реактивного состояния.

### Дополнительные действия
- Удалить временные bridge-модули, использовавшиеся для перехода между legacy и SFC.
- Убедиться, что widget registry, root app и lifecycle больше не зависят от legacy-loader.
- Зафиксировать окончательное state ownership для семейств `list`, `voc`, `split_button`, `table`.

### Ограничения
- один composable — одна предметная зона ответственности;
- нельзя делать composable, который знает про весь UI сразу;
- `table-specific` логика не должна утекать в общие composables без реальной необходимости;
- нельзя сохранять singleton-like реактивное состояние для пер-instance поведения виджетов.

### Результат этапа
Mixin-слой заменён на управляемую композицию, legacy bridge удалён, а новый runtime становится единственным центром инициализации.

## Этап 7. Асинхронный реестр виджетов и разбиение на чанки
### Цель
Сделать загрузку тяжёлых виджетов явной и ленивой.

### Изменения
- Заменить eager widgetFactory на async-реестр.
- Использовать `defineAsyncComponent` или эквивалентную схему с явными loader-функциями.
- Типизировать контракт `widget type -> component loader`.
- Выделить отдельные async chunks минимум для:
  - `table`
  - `list + voc + split_button`
  - `datetime`
- `debug` должен оставаться отдельным entry.
- Общие `core/common` модули не следует без необходимости разносить в лишние async-границы.

### Обязательные правила
- Не разрезать код "агрессивно" ради самого разрезания.
- Chunking делать по границам реального совместного использования.
- `list`, `voc`, `split_button` можно держать вместе только если это подтверждается фактическим графом использования и bundle output.
- Async loading должен применяться только там, где он реально уменьшает initial bundle.
- Базовая страница не должна становиться зависимой от лишних async-границ на `common`-слое.
- Нельзя допускать, чтобы один тяжёлый shared-helper затянул семейный chunk обратно в стартовую загрузку.
- Нужно отдельно проверить, не тянут ли `common`, `composables`, `table utils` и похожие shared-модули тяжёлые зависимости обратно в main entry.

### Результат этапа
Тяжёлые семейства не попадают в initial bundle по умолчанию, грузятся по требованию и не возвращаются скрыто в initial graph через shared imports.

## Этап 8. Ограниченный prefetch
### Цель
Подгружать только реально нужные тяжёлые чанки, а не все возможные.

### Правила
- После bootstrap страницы prefetch разрешён только для типов виджетов, которые действительно присутствуют в активных секциях.
- При загрузке данных модалки prefetch разрешён только для виджетов, которые реально потребуются этой модалке.
- Blanket-prefetch всех потенциальных виджетов запрещён.
- Нельзя допускать каскадной ленивой догрузки общих зависимостей при первом же базовом действии пользователя, если эти зависимости фактически нужны почти всегда.

### Результат этапа
Prefetch не превращается в скрытую полную загрузку всего UI и не ухудшает реальный путь первого взаимодействия.

## 9. Fallback, ошибки и гонки для async widgets

Для асинхронного реестра обязательно определить поведение на случай:
- неизвестного типа виджета;
- ошибки загрузки чанка;
- временной задержки загрузки;
- отсутствия компонента в конкретном `entry/runtime`-контексте;
- быстрого `mount/unmount` асинхронного виджета;
- закрытия модалки до завершения загрузки чанка;
- повторного открытия того же lazy widget;
- двух одинаковых lazy widget одновременно;
- ошибки чанка после частичной инициализации внутреннего состояния.

### Обязательное поведение
- ошибка одного виджета не должна валить весь root app;
- должен быть контролируемый fallback placeholder;
- ошибка должна логироваться;
- не должно быть бесконечного спиннера без диагностики;
- `unknown widget type` должен обрабатываться явно, а не молча;
- незавершённая асинхронная загрузка не должна оставлять утечки, подписки, таймеры или подвисшие placeholders;
- повторный `mount/unmount` не должен приводить к двойной инициализации;
- повторное открытие lazy widget должно возвращать UI в консистентное состояние.

## 10. Критерии завершения SFC-миграции

Миграция считается завершённой, когда одновременно выполнены все условия:

- `page` и `debug` монтируют root SFC;
- production-используемые Vue-компоненты переведены на `.vue`;
- inline `template:` больше не является основной production-моделью;
- глобальная регистрация не используется как основной способ сборки UI;
- mixin-слой заменён на `composables/utils`;
- временный legacy bridge удалён из production-path к концу этапа 6;
- SFC-слой не зависит от sequential legacy-loader как от центра инициализации;
- widget state имеет явного владельца и не течёт в неявные singleton-like reactive модули;
- новые composables, widget registry и async loaders имеют явный типизированный публичный контракт;
- тяжёлые семейства виджетов вынесены в отдельные async chunks;
- `page` не тянет `debug` в initial graph;
- shared-модули ведут себя одинаково в `page` и `debug`, если не задокументировано иное;
- тяжёлые чанки не входят в initial page bundle без реальной необходимости;
- initial graph не получает скрытый re-import тяжёлых зависимостей через `common/composables/utils`;
- CSS cascade и `theme/layout` contract не сломаны миграцией;
- проект проходит typecheck и build;
- smoke-проверка `page` и `debug` проходит без визуальных и behavioural regressions.

## 12. Риски

### Основные риски
- скрытая зависимость логики от текущего DOM или порядка инициализации;
- неявные runtime-зависимости, завязанные на глобальную регистрацию;
- расползание state ownership в `singleton-like` reactive модули;
- регрессии CSS cascade из-за спонтанных локальных стилей или `scoped styles`;
- визуальные и behavioural регрессии в тяжёлых виджетах;
- ложное улучшение bundle size за счёт ухудшения реального пути загрузки;
- гонки и утечки при async loading;
- перенос mixins в слишком крупные composables без реального упрощения системы.

### Как снижать риски
- начинать с обязательного аудита и инвентаризации;
- мигрировать по фиксированному порядку;
- после каждого этапа выпускать артефакт с остатками legacy и bridge-слоя;
- не совмещать структурный перенос и попутный функциональный рефакторинг без явной необходимости;
- отдельно проверять тяжёлые виджеты, async loading и page/debug parity;
- удалять временные bridge-решения строго по графику, не перетаскивая их в финальную архитектуру.

---

## 13. Итоговый результат

После завершения задачи фронтенд должен прийти в следующее состояние:

- `page` и `debug` работают через root `.vue` SFC;
- production-компоненты Vue живут в `.vue`;
- Composition API и `<script setup>` являются стандартной моделью для новых компонентов;
- legacy frontend runtime больше не скрыт под SFC и не является центром инициализации;
- mixin-слой заменён на composables и utility-модули;
- widget registry становится явным, типизированным и поддерживает async loading;
- тяжёлые виджеты вынесены в async chunks и грузятся по требованию;
- CSS cascade и `theme/layout` contract сохранены;
- widget state имеет явных владельцев и не течёт между экземплярами через скрытые singleton-like модули;
- initial page bundle уменьшен относительно baseline без ухудшения реального UX-пути загрузки.
