# Планировщик задач: спецификация для исполнителя

Документ из двух уровней:

1. **Уровень 1 — универсальный:** без привязки к Vue, таблице, `vm`. Планировщик опирается на него для любой задачи.
2. **Уровень 2 — примеры из репозитория** (виджет таблицы): только ориентир глубины; не подставлять эти термины в планы про API, бэкенд, статику без необходимости.

Доменные правила конкретного стека — в профильных skills (например [.ai_skills/skills_js.md](skills_js.md)).

---

## Уровень 1. Универсальные принципы

### 0. Классы рисков и обязанность плана

Ниже — **классы ошибок**, которые хороший план заранее «закрывает».

**Правило для планировщика:** для **каждого** пункта списка в теле плана должно быть **одно из двух**:

- **явный подраздел** (хотя бы 2–3 предложения), описывающий инвариант, гейты или осознанный отказ от покрытия в v1; **или**
- строка **`Не применимо`** + **одна причина** (почему этот класс риска к задаче не относится).

Иначе исполнитель прочитает список «для красоты» и не поймёт, что с ним делать.

**Контрпример (недостаточно):** «Не применимо: не нужно» / «не относится к задаче» без указания **какой** класс исключён и **почему** в этой задаче нет соответствующего риска (например: «нет async и общего снимка между слоями — только синхронный submit одной формы»). Такая строка не считается выполнением правила.

**Список классов:**

| Класс | Суть |
|--------|------|
| **Рассинхрон источников** | UI, локальное состояние, DOM и/или снимок события расходятся |
| **Гонки async** | Устаревшие данные после `await` без версии сессии или отмены |
| **Разные пути ввода** | Мышь, клавиатура, программный вызов ведут себя по-разному без основания |
| **Скрытые UX-решения** | Поведение «и так понятно» без записи в план |
| **Неявные инварианты** | Фокус, выделение, якорь навигации не описаны после мутации |
| **Краевые состояния** | Пустые данные, границы диапазона, режим редактирования |

### 1. Frontmatter плана (трекер / Cursor)

- **`name`**: короткое имя задачи.
- **`overview`**: одна строка — итог, ограничения, что нельзя забыть.
- **`todos`**: `id`, `content`, `status` (`pending` / `in_progress` / `completed`).

**`content` — критерий достаточности:** исполнитель по строке должен понять **что сделать в файле**, а не только «трогать» файл.

- **Недостаточно:** «обновить `table_widget.js`».
- **Достаточно:** «`table_widget.js` — добавить guard на `sessionId` перед вызовом paste» (или эквивалент без привязки к проекту: файл + **конкретное изменение** — guard, ветка, сигнатура, инвариант).

- **`isProject`**: по необходимости.

---

### 2. Цель и границы

- **Зачем** (продукт / техдолг / баг).
- **v1 vs вне scope** (отложенное — отдельная задача).
- **Уже существующие ограничения** среды (флаги, порядок загрузки, права) — чтобы не дублировать разведку в коде.

---

### 3. Архитектура и границы ответственности

Раздел **обязателен**, если срабатывает **хотя бы один** порог

- **три и более** затронутых модуля / файла верхнего уровня ответственности;
- **изменение публичного интерфейса** между слоями (экспорт, контракт API, события, общий предикат);
- **новая ветвь** в потоке данных между уже разделёнными слоями.

Если порог сработал:

- схема зависимостей (mermaid по желанию);
- таблица **модуль → одна ответственность**;
- явно: что **нельзя смешивать** (например, чистые билдеры конфигурации и побочные эффекты в одном слое).

---

### 4. Контракты данных и жизненный цикл

Когда состояние делится между слоями (UI, меню, транспорт, async):

- поля, типы, **семантика** каждого поля;
- **когда создаётся** снимок / контекст;
- **когда обнуляется**;
- правило при **рассинхроне версии** (no-op, закрытие UI, повторный запрос — что выбрано).

**Один предикат «можно ли действие»** для всех каналов ввода, если иначе получится «в одном месте можно, в другом нет».

#### Шаблон заполнения (скопировать в план и подставить значения)

Исполнителю проще дополнять **готовый каркас**, чем абстрактный чеклист.

```text
Снимок / контекст (имя): _______________

Поля (пример структуры, переименовать под задачу):
  версия / токен отмены: ___   // фиксирует момент или позволяет отбросить устаревший async-результат
  идентификатор сущности: ___ // при необходимости
  payload: { ... }             // только то, что нужно для действия

Создаётся:   при ____________________ (событие / открытие / keydown)
Обнуляется: при ____________________ (закрытие, Esc, уход со страницы — перечислить)
Правило при несовпадении версии / токена: ____________________
Закрывать UI при устаревшем вызове: да / нет — и почему
```

**Источник истины при нескольких каналах:** явный **приоритет** (например: зафиксированный снимок → состояние модели → fallback из DOM) и поведение при конфликте.

---

### 5. Поведение и UX: явные решения

- **Граничные случаи:** одно согласованное правило для «ничего не произошло, но жест наш / не наш» (например, отмена действия по умолчанию в браузере только после признания жеста своим).
- **Состояние после мутации:** что сбрасывается, что синхронизируется (выделение, фокус, «якорь» навигации — **назвать сущности задачи**, не копировать чужой стек). Если мутация **не влияет** на фокус / выделение / навигационный якорь — явно написать **`Не меняется`** (или одной фразой перечислить, что остаётся как было), чтобы было видно **осознанный** пропуск, а не пробел в плане.
- **Компромиссы v1:** пометка **осознанный компромисс v1**, если жертвуем контекстом ради простоты.
- **Намеренные ограничения:** **не баг для v1** + возможное смягчение позже.

**Неочевидный выбор:** формат «**Выбрано:** … / **Не выбрано:** … (почему)».

---

### 6. Ввод: клавиатура, события, среда

Если есть горячие клавиши или перехват событий:

- **порядок проверок** от дешёвых к дорогим;
- политика **`event.repeat`** по поджестам, если отличается;
- связь **`return true/false`** цепочки обработчиков с **`preventDefault`** (в JS `return false` сам по себе не отменяет действие браузера);
- **когда жест считается принадлежащим компоненту** (фокус внутри, флаг focus-within, валидное выделение — что уместно в задаче).

---

### 7. Алгоритмы, побочные эффекты, атомарность

- чувствительные **последовательности** (индексы, порядок splice, транзакции);
- **глубокое копирование** при необходимости: контракт + fallback, не «надеемся на shallow copy»;
- **фокус после асинхронного рендера:** число попыток, поведение при провале;
- **будущий undo:** одна пользовательская операция ↔ одна логическая мутация ↔ одна точка синхронизации с моделью/сервером (или явно иначе).

#### Атомарность — не только декларация

**Как исполнитель проверяет** после реализации (в плане указать, что сделать; конкретные имена методов — из задачи):

1. **Проход сценария:** одно пользовательское действие от события до сохранения состояния — не должно оставлять **промежуточного** состояния без явной спецификации в плане. Критерий промежуточности: **UI отображает данные, несовместимые ни с состоянием до операции, ни с состоянием после** (например, индекс строки уже сдвинут, а подпись/выделение ещё от старого набора).
2. **Поиск по изменённым файлам:** сколько раз для **одного** user gesture вызывается точка синхронизации (`onInput`, `emit`, `save`, `PATCH` — что в проекте принято); лишние вызовы — повод пересмотреть код.
3. **Тесты:** если есть — один сценарий ↔ один ожидаемый снимок состояния / один вызов мока API.

В плане для конкретной задачи можно сузить: например, «атомарность: не более одного `onInput` на пункт меню / один keydown-обработчик move».

#### Симметрия путей

Один **предикат доступности** и один **исполнитель** (общая функция/метод) для меню, клавиатуры и программного вызова — если все три существуют.

---

### 8. Меню и списки действий в UI

- различимые **иконки/метки** при похожих действиях;
- **порядок пунктов** как договорённость v1 + комментарий в коде при расширении;
- **разделители:** правила явно, не «на глаз»;
- **disabled** в UI **и** повторный guard в обработчике действия.

---

### 9. Файлы, порядок внедрения, документация

Нумерованный порядок: утилиты и контракты → ядро → UI → ввод → стили → обновление **соответствующего** skill/README, если меняется публичный контракт.

**Проверка по ходу:** после перевода каждого todo в **completed** исполнитель проверяет **инварианты из §0**, которые **этот шаг затрагивает** (ручной прогон или тесты — как указано в плане), а не откладывает всё на финал задачи.

### 10a. Чеклист: перед **написанием** плана

Короткий — чтобы сузить scope.

- [ ] Понятна **цель** и **граница v1**?
- [ ] Известны **затрагиваемые модули** (хотя бы список файлов/пакетов)? Если **нет** — сначала: короткая разведка по коду/поиск по репозиторию **или** первый todo с `id: explore-scope` и **минимальным** `content`, например: «Прочитать _X, Y, Z_ (точки входа / связанные модули) → в комментарии к todo или в теле плана: список файлов + список публичных интерфейсов / контрактов, которые затронет задача»; план не передавать исполнителю как «готовый», пока разведка не закрыта.
- [ ] Есть ли **async** — нужен ли снимок/версия?
- [ ] Есть ли **несколько путей ввода** — нужен ли общий предикат?
- [ ] Нужен ли подраздел по **§ 0** для каждого класса риска или «не применимо»?

---

### 10b. Чеклист: перед **передачей** плана исполнителю

Узкий финальный контроль (не раздувать — иначе модель отмечает галочки формально).

**§6, §7 (кроме атомарности) и §8** не обязательны на каждую задачу. Если соответствующая фича **есть** (хоткеи / перехват событий; мутации с последовательностями, копированием, фокусом после async; меню или списки действий) — в плане должен быть подраздел **или** «не применимо» с причиной. Если фичи **нет** — раздел не требуется, отдельная метка «не применимо» не нужна.

- [ ] **Overview + todos** с файлами; `content` по критерию §1.
- [ ] Нет вилок «А или Б» без выбранного варианта в тексте.
- [ ] **Контракт** заполнен шаблоном из §4 **или** явно «не нужен».
- [ ] **§0:** каждый класс риска закрыт подразделом или «не применимо».
- [ ] **§7 — атомарность:** указано, *как* проверить.
- [ ] **§6 / §7 (остальное) / §8:** по правилу выше — если релевантно, закрыто или «не применимо».
- [ ] **Порядок внедрения** и обновление документации указаны.

## Когда обновлять этот skill

Появился новый **стабильный класс задач** (формы, модалки, API) — вынести универсальный принцип в **уровень 1**. Детали стека — в **приложение** или в профильный skill, не размывать базовый уровень.

## Уровень 2 (приложение): ориентиры по проекту test_site — виджет таблицы

**Это не часть универсального уровня 1.** Не подставлять термины и файлы отсюда в планы про другие подсистемы без необходимости.

Пример **глубины** и лексики из планов `таблица_контекстные_меню_50f6b496` и `таблица_alt-строки_25c4af5f`. **Где взять полный текст:** в корне репозитория открыть **`.cursor/plans/`**, найти файл **по префиксу имени** в списке (например `таблица_контекстные_меню_`… или `таблица_alt-строки_`…; расширение обычно `.md`). Использовать при планировании **только** задач по этому виджету.

- **Слои:** `table_widget.js` (UI, open/hide), `table_context_menu.js` (`buildMenuItems`), `table_clipboard.js`, `table_keyboard.js`; снимок `contextMenuContext`, `sessionId`, paste с проверкой версии.
- **Жизненный цикл:** меню открыто → снимок живёт; `hideContextMenu` → `contextMenuContext = null`; новое открытие → новая сессия, старый async paste отменяется по `sessionId`.
- **Клавиатура:** `handleAltRowMoveDuplicate` до обязательного `getCellFromEvent` в общем диспетчере; `_tableFocusWithin`; ctx vs `selFocus`; duplicate игнорирует `repeat`, move — нет; `preventDefault` только после признания жеста «нашим».
- **Код:** `rowMoveDuplicateOpsAllowed`, `cloneTableRowDeep`, `focusSelectionCellWithRetry`; обновление [.ai_skills/skills_js.md](skills_js.md) при смене контракта.

Полные формулировки — только в этих файлах; сюда не переносить.
