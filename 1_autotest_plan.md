План создания фронтенд-автотестов
1. Цель и границы

Сделать отдельный автономный контур фронтенд-автотестов для проекта, который:

хранится в отдельной папке;
запускается своим скриптом из этой папки;
не смешивает тестовую инфраструктуру с основным frontend/runtime;
покрывает:
главную страницу;
переход с главной страницы на 2_widget_demo;
логику виджетов на странице 2_widget_demo;
демонстрационные table, включая сценарии по отдельным атрибутам.

На первом этапе в scope входят только frontend / browser tests.
Backend-логика, unit-тесты Python и интеграционные тесты сервера пока вне scope.

2. Базовое техническое решение
2.1. Выбранный стек
Фреймворк тестов: Playwright Test
Язык тестов: TypeScript
Тип тестов на первом этапе: E2E / UI integration
Браузер на старте: Firefox

Playwright сам по себе уже даёт runner, конфиг, retries, reporters, trace, CLI и multi-browser projects, поэтому не нужно собирать тестовый контур из нескольких разрозненных инструментов.

2.2. Почему не делать ставку на Selenium

Selenium использовать можно, но для вашего проекта это будет менее удачный старт:

у вас много интерактивного UI;
есть вкладки, меню, сложные виджеты и таблицы;
важна стабильность ожиданий и удобная отладка падений;
нужен быстрый ввод в работу без лишней инфраструктурной возни.

Поэтому Selenium здесь не запрещён, но не является лучшим первым выбором.

3. Требование к размещению тестов

Все тесты и всё, что к ним относится, должны лежать в отдельной папке, например:

frontend_tests/
  package.json
  playwright.config.ts
  tsconfig.json
  .gitignore
  README.md

  run.sh
  run_headed.sh
  open_report.sh

  tests/
    smoke/
    home/
    demo/
    widgets/
    tables/

  pages/
    home.page.ts
    demo.page.ts

  widgets/
    str.widget.ts
    text.widget.ts
    int.widget.ts
    float.widget.ts
    list.widget.ts
    voc.widget.ts
    table.widget.ts

  helpers/
    locators.ts
    yaml_loader.ts
    yaml_manifest.ts
    navigation.ts
    assertions.ts

  fixtures/
    manifests/
    snapshots/
    generated/

  reports/
3.1. Обязательные принципы
тесты не размазывать по основному frontend-коду;
у тестового контура свой package.json;
запуск делать из этой папки;
основной репозиторий должен воспринимать этот каталог как отдельный тестовый модуль.
4. Скрипты запуска

Внутри папки тестов должны быть свои скрипты.

Минимум:

run.sh — обычный headless-запуск;
run_headed.sh — запуск с видимым браузером;
open_report.sh — открыть HTML-отчёт;
при необходимости run_debug.sh — один тест в debug-режиме.

Playwright поддерживает CLI, HTML report и просмотр отчёта отдельной командой.

4.1. Что должен делать run.sh

Скрипт должен:

перейти в папку frontend_tests;
убедиться, что зависимости установлены;
поднять локальное приложение;
дождаться доступности URL;
запустить тесты;
сохранить отчёт.

Playwright умеет поднимать локальный сервер через webServer в конфиге, что как раз подходит для локального dev/runtime-запуска.

5. Архитектурный принцип тестов

Нельзя делать один огромный тест “открыли демо-страницу и пощёлкали всё подряд”.

Нужна трёхслойная схема:

5.1. Smoke layer

Проверяет, что:

приложение вообще стартует;
главная страница открывается;
навигация до 2_widget_demo работает;
демо-страница доступна.
5.2. YAML consistency layer

Проверяет соответствие UI и YAML:

какие страницы должны быть доступны;
какие вкладки/меню должны существовать;
какие виджеты должны быть выведены;
какие demo-блоки таблиц должны существовать.
5.3. Behavior layer

Проверяет уже поведение конкретных виджетов:

ввод;
выбор;
открытие/закрытие списков;
поиск;
мультивыбор;
модалки;
таблицы;
табы;
меню;
навигацию клавиатурой.
